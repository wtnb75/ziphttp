# AGENTS.md

このファイルは、ziphttp リポジトリで開発する人間とエージェントの共通ルールです。

## 1. 目的

- 変更を小さく安全に保つ
- パフォーマンスと可読性を両立する
- 配布バイナリと実運用挙動を壊さない

## 2. 対象範囲

- このファイルが置かれたディレクトリ配下すべて
- 特に Web 配信処理、圧縮処理、自己展開バイナリ関連の変更

## 3. 作業前チェック

- 既存の CLI オプションと互換性を維持する
- レスポンスヘッダーとステータスコードの後方互換を確認する
- キャッシュ系ヘッダー(ETag, Last-Modified)の条件分岐を維持する

## 4. 実装規約

### 4.1 基本方針

- 1 変更 1 目的を原則とする
- 既存の公開仕様を変える場合は README かテストを同時更新する
- 大きな設計変更より、段階的な改善を優先する

### 4.2 Go コードスタイル

- `gofmt` を前提に整形する
- エクスポートされる識別子は Go 標準命名に従う
- 略語は一貫性を優先する(例: URL, HTTP)
- ログは `slog` を使用し、キー名は短く意味が分かる語を使う

### 4.3 エラー処理

- エラーは握りつぶさず、呼び出し側で判断可能な形で返す
- ログ出力時は原因特定に必要な文脈(対象ファイル、メソッド、サイズなど)を添える
- 期待される分岐(例: 304 Not Modified)はエラー扱いしない

### 4.4 HTTP ハンドリング

- `Content-Type`, `Content-Encoding`, `Content-Length` の整合性を崩さない
- 条件付き GET の判定ロジックを変える場合は回帰テストを追加する
- 圧縮済みデータのパスは不要な再圧縮を避ける

### 4.5 並行処理と状態管理

- 共有状態は既存のロック戦略に合わせる
- ロックの粒度は必要最小限にし、I/O を長時間ロック内で実行しない
- 初期化と再読込の境界を明確に保つ

## 5. テスト規約

- 変更箇所に対応するテストを必ず追加または更新する
- 単体テストはテーブル駆動を基本とし、入力・期待値・検証観点をケースごとに明示する
- 正常系より先に境界値と異常系を洗い出し、コーナーケースを優先して実装する
- カバレッジ目標は 90% とし、少なくとも変更対象パッケージで 90% 未満にならないよう維持する
- 少なくとも以下を実行してから提案する
  - `go test ./...`
- 可能であれば以下も実行する
  - `task fmt`
  - `task lint`
  - `task test`

### 5.1 単体テストの書き方

- テスト名は振る舞いが分かる形にする(例: TestZipHandler_SendRaw_BrotliPreferred)
- 1 テスト 1 振る舞いを原則とし、失敗時に原因が特定しやすい粒度を保つ
- 時刻や乱数に依存する処理は注入または固定値を使い、再現性を確保する
- 比較対象が複雑な場合は差分が読みやすいアサーションを使う

### 5.2 コーナーケース重点方針

- HTTP 条件付き GET: If-None-Match / If-Modified-Since の境界と優先順
- 圧縮配信: gzip/deflate/br/zstd の選択分岐とフォールバック
- パス処理: 末尾スラッシュ、空パス、二重スラッシュ、`..` を含む不正パス
- ZIP 内容: 同名異メソッド、欠損エントリ、暗号化フラグ付きエントリ
- エラー処理: Open/OpenRaw 失敗時のステータスコードとログ文脈

### 5.3 カバレッジ運用

- 目標値確認の目安として以下を使用する
  - `go test ./... -coverprofile=cover.out -covermode=atomic -coverpkg=./...`
  - `go tool cover -func=cover.out`
- 合計カバレッジが 90% 未満の場合は、未到達分岐を特定してテストを追加する
- 目標未達のまま提案する場合は、未達理由と補完計画を変更説明に明記する

## 6. セキュリティ規約

- ZIP エントリ名の検証(例: `..` を含むパス)を弱めない
- 外部入力由来の値をヘッダーやログに出す場合は妥当性を意識する
- 依存ライブラリ更新時は脆弱性スキャン結果を確認する

## 7. パフォーマンス規約

- 配信時はコピー回数と再圧縮を増やさない
- 大きなファイルを扱う処理はメモリ使用量の増加に注意する
- 速度改善時は可観測な根拠(ベンチ、プロファイル、実測)を残す

## 8. 変更提案の出し方

- 変更理由を 1-3 行で説明する
- 影響範囲(機能、CLI、レスポンス、テスト)を明示する
- 未確認事項がある場合は前提条件として先に共有する

## 9. 禁止事項

- 無関係なリファクタリングを同じ変更に混ぜない
- 失敗時の挙動を黙って変更しない
- テストや lint の失敗を放置して完了扱いにしない

## 10. 推奨コマンド

- 形式整形と静的検査: `task fmt && task lint`
- テスト: `task test`
- ローカルビルド: `task build`

以上。
